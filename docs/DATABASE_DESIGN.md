# Database Design -- Smart Warehouse

**Author:** Misha Andalusia (001202400040)

## Overview

This document describes the data model used in the Smart Warehouse application. Right now the app uses mock data stored in JavaScript (no actual database), but the schema below represents what the data looks like and how the tables would be structured if we migrate to a real database like MongoDB or PostgreSQL.

---

## Tables

### 1. users

Stores all user accounts. Currently we have 4 demo users (one for each team member).

| Column | Type | Description |
|--------|------|-------------|
| id | VARCHAR | Primary key (e.g. "user-1") |
| name | VARCHAR | Full name |
| email | VARCHAR | Login email (e.g. risly@warehouse.io) |
| password | VARCHAR | Password (currently hardcoded as "admin123" for demo) |
| role | ENUM | admin / manager / operator |
| student_id | VARCHAR | University student ID |
| avatar | VARCHAR | 2-letter initials for avatar display (e.g. "RW") |

### 2. zones

Represents the 6 warehouse zones shown on the floor plan.

| Column | Type | Description |
|--------|------|-------------|
| id | VARCHAR | Primary key (e.g. "zone-a") |
| name | VARCHAR | Zone label (e.g. "Zone A -- Receiving") |
| description | VARCHAR | Short description of what the zone is for |
| color | VARCHAR | Hex color code for the zone on the map |
| x | INT | X position on the floor plan (percentage) |
| y | INT | Y position on the floor plan (percentage) |
| width | INT | Width on the floor plan (percentage) |
| height | INT | Height on the floor plan (percentage) |
| capacity | INT | Maximum capacity in units |
| used | INT | Currently used units |
| status | ENUM | active / warning |

### 3. cameras

There are 8 cameras placed across the warehouse zones.

| Column | Type | Description |
|--------|------|-------------|
| id | VARCHAR | Primary key (e.g. "cam-1") |
| name | VARCHAR | Display name (e.g. "Camera 1") |
| zone | VARCHAR | Foreign key to zones table |
| status | ENUM | online / offline |
| resolution | VARCHAR | Camera resolution (e.g. "1920x1080") |
| fps | INT | Frames per second (0 when offline) |

### 4. inventory

Stores all warehouse inventory items. Supports full CRUD operations.

| Column | Type | Description |
|--------|------|-------------|
| id | VARCHAR | Primary key (e.g. "INV-001") |
| name | VARCHAR | Item name |
| category | VARCHAR | One of: Electronics, Chemicals, Mechanical, Perishable, Automotive, Supplies, Safety, Logistics |
| zone | VARCHAR | Foreign key to zones table (e.g. "zone-a") |
| quantity | INT | Current stock count |
| min_stock | INT | Minimum stock threshold -- triggers "Low Stock" status when quantity drops below |
| status | ENUM | in-stock / low-stock / out-of-stock (auto-calculated based on quantity vs min_stock) |
| weight | FLOAT | Weight per unit in kg |
| last_detected | DATETIME | Last time the item was detected by the object detection system |

### 5. alerts

Notification alerts generated by the system. Supports severity filtering and read/unread state.

| Column | Type | Description |
|--------|------|-------------|
| id | VARCHAR | Primary key (e.g. "ALT-001") |
| type | ENUM | critical / warning / info |
| title | VARCHAR | Alert headline |
| message | TEXT | Detailed alert description |
| zone | VARCHAR | Foreign key to zones table (null for system-wide alerts) |
| read | BOOLEAN | Whether the user has read this alert |
| timestamp | DATETIME | When the alert was created |

### 6. activity_log

Audit trail that records every action in the system. Some entries are created automatically by the reducer (CRUD operations) and some are injected by the LiveEventSimulator.

| Column | Type | Description |
|--------|------|-------------|
| id | VARCHAR | Primary key (e.g. "LOG-001") |
| user | VARCHAR | Name of the user who performed the action, or "System" for automated events |
| action | VARCHAR | What happened (e.g. "Added new item", "Object detected") |
| target | VARCHAR | What was affected (e.g. "INV-003 (Steel Bearings Box)") |
| type | ENUM | create / update / delete / detection / alert / export / system / approval |
| timestamp | DATETIME | When it happened |

### 7. object_classes

Reference table for the types of objects the detection system can identify.

| Column | Type | Description |
|--------|------|-------------|
| id | INT | Primary key |
| name | VARCHAR | Object class name (e.g. "Cardboard Box", "Forklift") |
| color | VARCHAR | Hex color used for the bounding box |
| icon | VARCHAR | Emoji icon for display |

### 8. detection_stats

Aggregated statistics for the detection system. Displayed on the Dashboard page.

| Column | Type | Description |
|--------|------|-------------|
| today | INT | Number of detections today (1,247) |
| yesterday | INT | Number of detections yesterday (1,103) |
| week | INT | Total detections this week (8,451) |
| month | INT | Total detections this month (34,209) |
| accuracy | FLOAT | Model accuracy percentage (97.3%) |
| avg_processing_time | INT | Average inference time in milliseconds (42ms) |
| cameras_online | INT | Number of cameras currently online (7) |
| cameras_total | INT | Total number of cameras (8) |

## Entity Relationships

```
users ──< activity_log      (one user has many log entries)
zones ──< cameras            (one zone has many cameras)
zones ──< inventory          (one zone has many inventory items)
zones ──< alerts             (one zone has many alerts)
cameras ──< detections       (one camera produces many detections)
```

In short:
- Each camera belongs to a zone
- Each inventory item is stored in a zone
- Each alert can be linked to a zone (or null for system-wide alerts)
- Each activity log entry is linked to a user

## Current Implementation

Since this is a frontend-only application, the data is stored in:

- **mockData.js** -- Static initial data for all tables
- **WarehouseContext.jsx** -- Runtime state managed by useReducer
- **AuthContext.jsx** -- User session state

The reducer handles these operations:
- `ADD_INVENTORY_ITEM` -- Adds item and auto-creates an activity log entry
- `UPDATE_INVENTORY_ITEM` -- Updates item and auto-logs the change
- `DELETE_INVENTORY_ITEM` -- Removes item and auto-logs the deletion
- `MARK_ALERT_READ` / `MARK_ALL_ALERTS_READ` -- Toggles alert read status
- `ADD_TOAST` / `REMOVE_TOAST` -- Manages toast notifications
- `ADD_ACTIVITY` -- Manually adds an activity log entry (used by LiveEventSimulator)
- `TOGGLE_SIDEBAR` -- UI state for sidebar collapse
- `UPDATE_ZONE` -- Updates zone data
